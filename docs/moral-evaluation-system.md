# 德育评价系统说明

## 版本信息

- 适用系统：班级管理系统 v3.1.1
- 整理日期：2026-04-28
- 来源：由原 `德育评价系统设计方案.md`、`德育评价系统合并方案.md`、`德育评价系统合并实施计划.md` 合并整理

## 一、定位

德育评价系统是班级管理系统中的德育业务模块，目标是把学生基础数据、班级数据、日常表现、校级事件、处分、德育任务、生日提醒、学生画像和 AI 诊疗纳入统一的数据和权限体系。

核心目标：

- 数据库暂不变更：继续使用 SQLite，德育数据落在 `lesson/databases/moral.db`，Excel 仅作为基础数据导入来源。
- 角色权限统一：使用 admin、jiaowu、xuefa、cleader、teacher、student、parent 七类角色。
- 德育评价可计算：以学期为评价周期，支持个人、班级和年级维度统计。
- 现有功能兼容：保留班级作业、公告、请假、文件管理、课表等既有功能。

## 二、业务模型

### 2.1 功能模块

| 模块 | 说明 |
|------|------|
| 数据输入 | 录入日常表现、校级事件、处分、德育任务完成情况和集体事件 |
| 数据分析 | 统计学生、班级、年级的事件、分数、排名和趋势 |
| 评价赋分 | 基于基础分和各类记录计算德育总分、等级和预警 |
| 数据展示 | 查询个人报告、班级汇总、年级汇总，支持导出 |
| 扩展能力 | 生日提醒、学生画像、AI 诊疗、违纪累进规则 |

### 2.2 数据输入来源

| 来源 | 内容 | 分值影响 |
|------|------|----------|
| 日常常规表现 | 积极事件、消极事件 | 加分或扣分 |
| 校级及以上事件 | 荣誉、违纪、处分 | 加分或扣分 |
| 年级德育任务 | 按级号设置的必修或选修任务 | 完成后加分 |
| 集体事件 | 班级荣誉、集体活动、集体违纪 | 面向班级成员批量分配 |

### 2.3 计分规则

```
德育总分 = 基础分 + 日常表现分 + 校级事件分 + 任务完成分 + 集体事件分 + 处分调整分
```

默认等级：

| 等级 | 分数范围 |
|------|----------|
| 优秀 | >= 90 |
| 良好 | 80-89 |
| 合格 | 60-79 |
| 不合格 | < 60 |

说明：

- 基础分和等级线应读取 `moral_config`，不要硬编码。
- 德育总分允许超过 100 分。
- 已生成或已确认的历史评价应保留当时快照，避免配置变更影响历史解释。

## 三、核心规则

### 3.1 学期与历史

| 规则 | 说明 |
|------|------|
| 学期隔离 | 日常表现、校级事件、评价结果均关联学期 |
| 历史保留 | 学期切换不删除历史记录 |
| 跨学期查询 | 支持按学生、班级、级号聚合历史趋势 |
| 当前学期 | 录入和默认统计使用唯一当前学期 |

### 3.2 德育任务结转

| 规则 | 说明 |
|------|------|
| 任务周期 | 以学年为主，可配置学期截止、学年截止、无截止 |
| 已完成任务 | 分值保留在完成学年或完成学期 |
| 未完成任务 | 可在学年末结转 |
| 结转分值 | 每结转一次按配置比例递减，历史方案建议 60% |
| 最大结转次数 | 默认 2 次 |
| 超限处理 | 作废或不再计分 |

示例：30 分任务结转两次后完成，得分为 `30 * 0.6 * 0.6 = 10.8`。

### 3.3 集体事件

集体事件用于批量处理班级层面的荣誉、活动和违纪。系统创建集体事件后，应按事件发生时的班级成员快照生成分配记录，避免后续调班影响历史归属。

处理流程：

1. 创建集体事件。
2. 选择级号、班级、事件类型、分值和发生日期。
3. 系统为目标学生生成分配记录。
4. 支持标记未参与或撤销分配。
5. 评价计算读取有效分配记录。

### 3.4 学生状态与班级调整

| 状态 | 德育记录 | 参与统计 | 任务处理 |
|------|----------|----------|----------|
| 在校 | 正常记录 | 参与 | 正常进行 |
| 休学 | 暂停新增 | 不参与 | 任务冻结 |
| 复学 | 恢复记录 | 参与 | 解冻并继续 |
| 转入 | 开始记录 | 参与 | 新分配任务 |
| 转出 | 停止记录 | 不参与 | 未完成任务作废或归档 |
| 毕业 | 历史可查 | 不参与当前统计 | 未完成任务作废 |

班级变化应写入 `student_class_history`，学籍状态变化应写入 `student_status_change`。留级、调班、复学、转入、转出和毕业都应可追溯。

### 3.5 处分与违纪

处分记录应支持创建、更新、撤销和查询。违纪累进规则由 `violation_escalation_rule` 管理，用于根据同类或同周期违纪次数给出加重建议。

撤销处分时不要删除原记录，应保留撤销时间、撤销原因和操作人。

### 3.6 数据可见性

| 角色 | 默认可见范围 |
|------|--------------|
| admin | 全部数据 |
| jiaowu | 教学和基础数据，全校统计 |
| xuefa | 德育数据，全校统计 |
| cleader | 本班学生和本班统计 |
| teacher | 自己录入的记录和授权范围内学生 |
| student | 本人数据 |
| parent | 子女数据 |

## 五、数据模型

### 5.1 表分组

| 分类 | 表 | 说明 |
|------|----|------|
| 基础数据 | `school_year`、`semester`、`grade`、`grade_level_config`、`class`、`student`、`teacher`、`role` | 学校组织、账号和角色基础 |
| 学籍履历 | `student_class_history`、`student_status_change` | 班级变动和学籍状态变更 |
| 德育核心 | `daily_event_type`、`school_event_type`、`grade_moral_task`、`student_daily_record`、`student_school_record`、`student_task_finish`、`punishment_record`、`collective_event`、`collective_event_distribution`、`moral_evaluation` | 评价计算的主要数据 |
| 扩展功能 | `student_profile`、`student_profile_history`、`profile_config`、`ai_consultation`、`ai_consultation_message`、`birthday_reminder`、`birthday_reminder_config` | 画像、诊疗和生日提醒 |
| 配置日志 | `violation_escalation_rule`、`semester_carryover_config`、`data_visibility_config`、`warning_config`、`warning_log`、`task_carryover_log`、`moral_operation_log`、`moral_config` | 规则、预警、结转、审计和系统参数 |

### 5.2 关键约束

- 学生学号、教师账号、班级编码应保持唯一。
- 每次只能有一个当前学期。
- 德育记录必须关联学生、学期、事件类型和录入人。
- 集体事件分配应关联集体事件和学生，并保留是否有效。
- 所有影响分值的记录应保留分值快照。
- 软删除或撤销优先于物理删除，便于审计。

## 六、API 模块

| 模块 | 路径前缀 | 说明 |
|------|----------|------|
| 基础管理 | `/api/moral/admin` | 级号、班级、学生、学年学期、配置、日志 |
| 日常表现 | `/api/moral/daily-records` | 日常事件类型和日常记录 |
| 校级事件 | `/api/moral/school-records` | 校级事件类型和记录 |
| 德育任务 | `/api/moral/tasks` | 任务配置和完成记录 |
| 处分管理 | `/api/moral/punishments` | 处分记录和撤销 |
| 评价查询 | `/api/moral/evaluations` | 学生、班级、级号评价计算和查询 |
| 学生画像 | `/api/moral/profiles` | 画像生成、查询和配置 |
| 生日提醒 | `/api/moral/birthdays` | 生日列表、提醒记录和提醒生成 |
| AI 诊疗 | `/api/moral/consultations` | 诊疗会话和消息 |

API 设计要求：

- 所有写操作校验权限并写入 `moral_operation_log`。
- 批量导入接口先校验再落库，返回成功数、失败数和失败原因。
- 评价计算接口应支持按学生、班级、级号、学期维度查询。
- 历史评价结果应可重复查询，必要时支持重新计算。

## 七、数据库与合并原则

### 7.1 现有系统来源

| 数据 | 原来源 | 当前策略 |
|------|--------|--------|
| 教师 | `checkTemplate.xlsx` | 导入或同步到 SQLite `teacher` |
| 学生 | `checkTemplate.xlsx` | 导入或同步到 SQLite `student` |
| 班级 | `checkTemplate.xlsx` | 导入或同步到 SQLite `class`、`grade` |
| 德育数据 | SQLite `lesson/databases/moral.db` | 保持 SQLite，不切换数据库 |
| 请假 | SQLite `inout.db` | 保留现有接口 |
| 作业公告 | SQLite `homework.db` | 保留现有接口 |
| 认证 | JWT + Excel/配置 | JWT + SQLite 角色权限 |

### 7.2 初始化与同步步骤

1. 备份 `lesson/databases/*.db`、`lesson/moral.db` 和 `checkTemplate.xlsx`。
2. 使用 `python scripts/create_moral_tables.py --sqlite` 创建或补齐德育 SQLite 表结构。
3. 从 Excel 导入或同步级号、班级、学生、教师基础数据。
4. 生成学生班级履历和初始学籍状态。
5. 初始化学年、学期、事件类型、系统参数。
6. 运行 `python scripts/verify_moral_system.py` 做数据完整性校验。
7. 后端 API 继续通过 `utils.sqlite_moral_db` 访问德育数据。

### 7.3 验证清单

- 源数据和目标数据数量一致。
- 班级、学生、教师的唯一标识无冲突。
- 学生均有关联班级和级号。
- 当前学期唯一。
- 外键约束、分值范围、状态枚举生效。
- 现有前端调用兼容。

## 九、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 数据初始化失败 | 高 | 初始化前备份 SQLite 和 Excel，脚本支持重复执行或手动回滚 |
| API 兼容性问题 | 中 | 保留旧接口路径，维持 SQLite 访问层稳定 |
| 权限配置错误 | 高 | 权限矩阵测试，关键操作审计 |
| 评价计算不一致 | 高 | 分值快照、可复算接口、样例数据对账 |
| 性能不足 | 中 | 为学生、班级、学期、事件类型建立索引 |
