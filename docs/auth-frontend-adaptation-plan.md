# 鉴权配置前端适配方案

## 1. 设计目标

前端权限配置页不能只展示“接口路径 + 角色复选框”，而要让管理员准确理解：

1. 这条 API 操作什么业务对象
2. 谁可以调用
3. 谁能看哪些记录
4. 谁能给哪些对象创建
5. 谁能改、删、复核、导出哪些记录

最终目标是：

> 配置者不需要读代码，也能正确修改权限。

## 2. 当前页面已有能力

现有 `ApiPermission.vue` 已支持：

- 按模块筛选 API
- 编辑 API 路径、方法、匹配方式
- 编辑角色、等级、策略
- 编辑 `data_scope_rules`
- 编辑 `target_scope_rules`
- 应用模块权限
- 同步默认范围
- 套用常用矩阵

这说明底座已经具备，不需要推倒重做。

## 3. 当前页面的主要问题

### 3.1 仍以“接口字段”为中心

管理员看到的是：

- `/api/moral/daily-records/update`
- `PUT`
- 一堆角色和范围

但他更需要看到：

- 这是“日常表现”
- 动作是“编辑记录”
- 数据对象是“日常表现记录”
- 推荐规则是“班主任和年级主任只改自己创建”

### 3.2 数据范围与动作范围语义不够分开

当前页面只有：

- 数据范围
- 目标范围

但编辑 / 删除本质上属于“动作范围”，与列表查看不是同一个问题。

### 3.3 多角色叠加不够直观

例如戴建海老师同时是：

- 班主任
- 年级主任

管理员很难从当前界面直接预判最终效果：

- 查看范围会并集
- 创建目标会并集
- 编辑删除仍可能只是 `own_created`

### 3.4 风险操作缺少校验和预览

当前容易发生：

- 允许教师访问 API，但未填教师范围
- 创建接口配置了 `data_scope_rules`，却忘了 `target_scope_rules`
- 编辑接口把班主任配成 `managed_classes`，导致可以改别人记录
- 动态路由忘记 `pattern`

## 4. 推荐的信息架构

建议把权限管理页升级为 3 层。

### 4.1 第一层：模块总览

左侧保留模块树，但补充统计：

- API 数量
- 已强制鉴权数量
- 缺范围配置数量
- 风险配置数量

示例：

```text
日常表现
12 APIs
10 已完整配置
2 待补齐
```

### 4.2 第二层：API 配置列表

表格建议新增列：

| 列 | 说明 |
| --- | --- |
| 业务对象 | 例如：日常表现记录 |
| 动作类型 | 查看 / 创建 / 编辑 / 删除 / 导出 / 复核 |
| 鉴权模式 | JWT / 微信 / 双通道 |
| 记录范围 | 已配置摘要 |
| 目标范围 | 已配置摘要 |
| 动作范围 | 已配置摘要 |
| 风险状态 | 完整 / 缺失 / 冲突 |

其中：

- `业务对象 = resource_type`
- `动作类型 = action_type`
- `动作范围 = operation_scope_rules`

### 4.3 第三层：结构化编辑抽屉

不再用“长表单对话框”堆所有字段，改为右侧抽屉或宽弹窗，分 5 个 Tab：

1. 基本信息
2. 调用权限
3. 查看范围
4. 创建目标
5. 动作范围

## 5. 各 Tab 设计

### 5.1 基本信息

字段：

- API 名称
- API 路径
- HTTP 方法
- 匹配方式
- 所属模块
- 业务对象
- 动作类型

建议：

- `resource_type` 用下拉选择
- `action_type` 用下拉选择
- 对已有模块提供自动识别和智能填充

### 5.2 调用权限

字段：

- 公开访问
- 是否参与后端鉴权
- 鉴权通道：
  - JWT
  - 微信 token
  - 双通道
- 是否继承模块
- 允许角色
- 最低等级
- 鉴权策略

交互建议：

- 选择“公开”后，隐藏角色与等级
- 选择“继承模块”后，角色与等级只读
- 选择 `enforce_backend=0` 时，显示醒目标识：
  - “该 API 不受统一后端权限控制”

### 5.3 查看范围

仅在 `action_type` 属于以下类型时显示：

- list
- detail
- search
- summary
- trend
- export

展示方式：

| 角色 | 可查看范围 |
| --- | --- |
| 教师 | 自己创建 |
| 班主任 | 自己创建、管理班级 |
| 年级主任 | 自己创建、管理年级 |
| 学发 | 全校 |

建议保留 checkbox 形式，但加：

- 业务模板一键套用
- 风险提示
- 多角色预览

### 5.4 创建目标

仅在以下动作显示：

- create
- batch-create
- import
- assign

展示：

| 角色 | 可操作目标 |
| --- | --- |
| 教师 | 任教班级 |
| 班主任 | 任教班级、管理班级 |
| 年级主任 | 任教班级、管理年级 |
| 学发 | 全校学生 |

### 5.5 动作范围

这是建议新增的核心配置区。

用于：

- update
- delete
- revoke
- review
- close
- export

字段建议：

```json
{
  "teacher": ["own_created"],
  "cleader": ["own_created"],
  "g_leader": ["own_created"],
  "xuefa": ["all"],
  "admin": ["all"]
}
```

前端展示仍然是角色 x 范围矩阵。

## 6. 配置模板

为了“准确 + 高效”，不能让管理员每次手工配。

建议提供模板库。

### 6.1 日常记录类模板

适用：

- 日常表现
- 点滴记录

模板内容：

- 查看范围：
  - teacher = own_created
  - cleader = own_created + managed_classes
  - g_leader = own_created + managed_grades
  - xuefa/jiaowu/admin = all

- 创建目标：
  - teacher = teaching_classes
  - cleader = teaching_classes + managed_classes
  - g_leader = teaching_classes + managed_grades
  - xuefa/jiaowu/admin = all_students

- 动作范围：
  - teacher/cleader/g_leader = own_created
  - xuefa/jiaowu/admin = all

### 6.2 学发专管模板

适用：

- 校级事件
- 处分管理
- 德育任务
- 集体事件

模板内容：

- 调用角色：xuefa/admin
- 查看范围：all
- 创建目标：all_students 或 all
- 编辑删除：all

### 6.3 学生档案查看模板

适用：

- 一生一册
- 学生画像
- 学生趋势图

模板内容：

- cleader = managed_classes
- g_leader = managed_grades
- xuefa/jiaowu/admin = all

## 7. 风险校验规则

前端保存前应做静态校验。

### 7.1 缺失配置

- `action_type=create` 但没有目标范围
- `action_type=list/detail/trend` 但没有查看范围
- `action_type=update/delete` 但没有动作范围

### 7.2 明显冲突

- 允许角色包含 `teacher`，但范围未配置 teacher
- `teacher` 编辑范围配置为 `managed_classes`
- `g_leader` 查看记录却未配置 `managed_grades`
- 动态路径含 `{id}`，但匹配方式不是 `pattern`

### 7.3 危险配置提醒

- `allow_missing=True` 的接口
- `enforce_backend=0`
- `is_public=1`
- `allowed_roles=[] + min_level=0`

这些都应在表格层有明显 badge。

## 8. 多角色效果预览

建议新增“权限效果预览”面板。

管理员可选择：

- 角色组合：
  - teacher
  - cleader
  - g_leader
  - cleader + g_leader

系统展示合并后的结果：

```text
戴建海老师（班主任 + 年级主任）

查看：
- 自己创建
- 管理班级
- 管理年级

创建目标：
- 任教班级
- 管理班级
- 管理年级

编辑删除：
- 自己创建
```

这能极大减少配置理解成本。

## 9. 推荐的页面结构

建议最终布局：

```text
左侧：模块树
中间：API 列表
右侧：配置抽屉 / 详情面板
```

API 列表支持：

- 搜索路径
- 搜索业务对象
- 按动作类型筛选
- 按配置状态筛选
- 批量应用模板
- 批量设置角色

## 11. 当前页面落地情况

截至 2026-05-13，`moral/config/api-permission` 已支持：

- 左侧模块树 + 风险数量
- API 搜索、资源类型筛选、动作类型筛选
- 查看 / 创建 / 动作范围配置
- 预置模板套用
- 多角色预览校验
- 风险审计与跳转定位
- 审计报告导出
- 批量设置允许角色

下一步更偏体验优化，而不是能力缺口：

- 对“同一资源下的一组 API”增加更高层的批量规则编排

## 12. 与当前后端实现的对齐结论

截至 2026-05-13：

- 前端页面配置的允许角色、数据范围、目标范围、动作范围，均会被后端实时读取。
- dashboard 趋势、任务结转、菜单权限、数据库管理、基础配置、累进规则学生视图等接口，已可通过页面配置并参与风险巡检。
- 页面上的“预览校验”用于帮助管理员理解多角色叠加效果；最终生效仍以后端数据库配置判断为准。
- 当前权限审计结果为 `174` 条 API、`0` 条风险。
